# Supabase Security Scenarios

This document describes a few real-world security scenarios that frequently appear in Supabase-based applications.

The examples focus on multi-tenant systems using:

- Supabase Auth
- Row Level Security (RLS)
- Edge Functions
- Supabase Storage

These scenarios illustrate how small implementation mistakes can lead to authorization failures or tenant isolation issues.

---

# Scenario 1 — RLS Policy Identity Mismatch

## What happened

A multi-tenant application used Row Level Security to restrict access to tenant data.

Users reported that they could not see their own records even though they were authenticated.

The application appeared to work correctly for administrators but failed for regular users.

---

## Root cause

The RLS policy compared the wrong identity fields.

Example policy:

```sql
profiles.id = auth.uid()
```

In the actual data model, the relationship was:

```sql
profiles.user_id = auth.uid()
```


Because the wrong column was used, the policy always evaluated to false.

---

## Result

The database returned empty result sets even for valid users.

This created confusing behavior where:

- authentication succeeded
- queries executed normally
- but no rows were returned

---

## Fix

Update the policy to reference the correct identity column:

```sql
profiles.user_id = auth.uid()
```


After correcting the identity comparison, the RLS policy enforced access correctly.

---

# Scenario 2 — Edge Function Bypassing RLS

## What happened

An Edge Function was responsible for returning tenant-specific data.

Users could only access the function after authentication.

However, the function returned records belonging to other tenants.

---

## Root cause

The Edge Function created a Supabase client using the `service_role` key.

Example pattern:

```sql
createClient(SUPABASE_URL, SERVICE_ROLE_KEY)
```


Because `service_role` bypasses RLS, the database returned rows without tenant filtering.

---

## Result

The function unintentionally exposed cross-tenant data.

This type of issue can be difficult to detect because:

- the API endpoint appears to work
- authentication is present
- but the database query runs with full privileges

---

## Fix

Create the Supabase client using the anon key and forward the user JWT.

Example pattern:

```sql
createClient(
SUPABASE_URL,
ANON_KEY,
{ global: { headers: { Authorization: req.headers.get("Authorization") } } }
)
```


This ensures that PostgreSQL evaluates RLS policies using the correct user identity.

# Scenario 3 — Signed URL Leak

## What happened

A Supabase Storage bucket was configured as private.

Files were accessed through signed URLs generated by an Edge Function.

Users reported that they could download files belonging to other tenants.

---

## Root cause

The Edge Function generated signed URLs without validating tenant membership.

The function assumed that the file path itself enforced isolation.

Example object path:

```text
<family_id>/photo.jpg
```


However, the function never verified that the requesting user actually belonged to the family.

---

## Result

Users could request signed URLs for files belonging to other tenants if they knew or guessed the path.

---

## Fix

Before generating a signed URL, the server must verify tenant membership.

Typical verification pattern:

```sql
EXISTS (
SELECT 1
FROM family_members
WHERE family_members.family_id = <family_id>
AND family_members.user_id = auth.uid()
)
```


Only after successful verification should the signed URL be generated.

---

# Summary

These scenarios highlight a common theme in Supabase security reviews:

Authorization failures rarely come from a single obvious mistake.

They usually arise from small gaps between:

- authentication
- identity propagation
- RLS policy design
- server-side logic

Security reviews should therefore focus on validating the entire request path rather than inspecting configuration in isolation.
